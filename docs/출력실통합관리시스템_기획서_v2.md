## 1. Context — 왜 만드는가

### 1.1 현재 운영의 문제점

- **잉크젯 출력**: 담당자가 B2에서 JPG를 수동으로 찾아서 다운로드 → 비효율
- **출력 이력 부재**: 누가, 언제, 어떤 파일을 다운받아 출력했는지 기록 없음
- **상태 가시성 부족**: print-queue 텍스트 목록만 있어 한눈에 파악 어려움
- **임포지션 수동 작업**: 할로겐 RIP에서 담당자가 매번 NUP·여백 설정
- **설정 오류 위험**: 규격이 많아 잘못 설정 시 용지·잉크 손실 발생

### 1.2 목표

출력담당자가 "출력실" 한 화면에서 모든 작업을 처리하고, 임포지션부터 핫폴더 전달까지 자동화되며, 모든 이력이 자동 기록되는 통합 시스템을 구축한다.

### 1.3 통합 효과

| 영역 | 개선 전 | 개선 후 |
|------|---------|---------|
| 파일 준비 | 수동 다운로드 | B2 자동 복사 |
| 임포지션 | RIP에서 수동 설정 | ERP 자동 PDF 생성 |
| 상태 관리 | 텍스트 목록 | Kanban 시각화 |
| 이력 추적 | 없음 | 자동 기록 |
| 파일 전달 | 담당자 USB | 핫폴더 자동 |
| 에러 대응 | 재출력 반복 | 실패 알림 + 재처리 |

## 2. 전체 시스템 아키텍처

PhotoCafe ERP 기존 스택 위에 두 개의 신규 모듈(임포지션, 출력실)이 추가됩니다.

### 2.1 신규 모듈 구조

| 계층 | 구성요소 | 역할 |
|------|----------|------|
| DB | OrderItem 확장 + 4개 신규 테이블 | 상태/이력/프리셋 저장 |
| Queue | BullMQ ImpositionQueue | 백그라운드 PDF 생성 |
| Service | ImpositionService + PrintRoomService | 면배치, 핫폴더 전달 |
| Storage | B2 print-ready/ 폴더 | 생성된 PDF + 원본 JPG |
| API | /imposition + /print-room | REST 엔드포인트 |
| UI | /print-room Kanban | 출력 담당자 화면 |
| Agent | 출력 PC 에이전트 | B2 → 핫폴더 자동 복사 |

### 2.2 데이터 흐름 한눈에 보기

```
주문 확정          B2 복사            임포지션           핫폴더          할로겐 RIP        SmartStream
(검수 완료)   →   (print-ready/)  →  (PDF 자동생성)  →  (에이전트)  →   (컬러프로파일)  →  (Indigo 출력)
```

## 3. 확정된 출력 워크플로우

### 3.1 인디고 라인 (HP Indigo)

```
B2 JPG  →  ERP 임포지션 PDF  →  핫폴더 자동저장  →  할로겐 RIP (CMYK)  →  SmartStream  →  Indigo 출력
```

> **핵심**: ERP가 면배치(임포지션)된 PDF를 생성하여 할로겐 RIP 핫폴더로 자동 전달. 할로겐 RIP은 CMYK 컬러 프로파일만 입혀서 SmartStream으로 보냄.

### 3.2 잉크젯 라인

```
B2 JPG  →  핫폴더 자동저장  →  출력 프로그램 (JPG 인쇄)  →  잉크젯 출력
```

> **핵심**: 잉크젯은 PDF 변환 불필요. JPG 원본을 그대로 잉크젯 출력 PC 핫폴더로 자동 전달.

### 3.3 출력 방식별 비교

| 구분 | 인디고 | 잉크젯 |
|------|--------|--------|
| 파일 형식 | PDF (임포지션 완료) | JPG 원본 |
| ERP 처리 | 임포지션 + 크롭마크 | 복사만 |
| 컬러 변환 | 할로겐 RIP 담당 | 출력 프로그램 담당 |
| 색상공간 | RGB → CMYK (RIP) | sRGB / AdobeRGB |
| 해상도 | 300dpi | 240dpi / 360dpi |
| 핫폴더 경로 | `\\printpc\hotfolder\indigo\` | `\\printpc\hotfolder\inkjet\` |

## 4. 임포지션 PDF 출력 스펙

포토미 출력 환경, 할로겐 RIP, HP SmartStream과 협의를 통해 확정된 PDF 사양입니다.

### 4.1 PDF 기본 스펙

| 항목 | 값 |
|------|-----|
| **블리드(재단여백)** | **없음 — 실사이즈 출력** |
| **크롭마크** | **포함 (CropMark)** |
| **크롭마크 색상** | **CMYK K100 (C:0 M:0 Y:0 K:100)** |
| 크롭마크 길이 | 5mm (표준) |
| 크롭마크 두께 | 0.25pt (표준) |
| 크롭마크 오프셋 | 면 경계에서 2mm 떨어진 위치 시작 |
| 출력 해상도 | 300dpi |
| 색상공간 | RGB (CMYK 변환은 할로겐 RIP) |
| PDF 버전 | PDF 1.4 |
| MediaBox | 실사이즈 + 크롭마크 여백 10mm |
| 임포지션 표준 | 독자 방식 (JDF는 향후 검토) |

### 4.2 NUP별 임포지션 규칙

| NUP | 적용 규격 예시 | 용지방향 | 면 배치 | 면/장 |
|------|---------------|----------|---------|-------|
| 16up | 3×4, 4×3 | 가로 | 4열×4행 | 16 |
| 8up | 6×4, 4×6 | 가로 | 4열×2행 | 8 |
| 4up | 5×7, 7×5, 6×8 외 | 가로 | 2열×2행 | 4 |
| 2up | 8×10, A4 외 | 가로/세로 | 2열×1행 | 2 |
| 1up | 10×10, 11×11 | 세로 | 1면 | 1 |
| 1+up | 14×11, 11×14 외 | 세로 | 1면(대형) | 1 |
| 1++up | 14×16, 16×14 | 세로 | 1면(특대) | 1 |

### 4.3 면 배치 좌표 계산

- 용지 크기 = 실사이즈 × NUP 열수(가로) × NUP 행수(세로) + 양쪽 10mm 여백
- 각 면 X = 여백 + (열 인덱스 × 면너비)
- 각 면 Y = 여백 + (행 인덱스 × 면높이)  ※ pdf-lib은 좌하단 기준
- 면 간격(거터) = 0mm (면 밀착, 재단선으로 구분)
- 면 순서 = 좌→우, 위→아래 (한국 출력 표준)

### 4.4 크롭마크 그리기 로직 (pdf-lib)

각 면의 4모서리에 8개의 라인(모서리당 수평+수직)을 그립니다.

```javascript
// 각 면 (x, y, w, h)의 좌상단 모서리 예시
drawLine({ start:{x:x-7, y:y+h+2}, end:{x:x-2, y:y+h+2}, color:rgb(0,0,0), thickness:0.25 })
drawLine({ start:{x:x+2, y:y+h+2}, end:{x:x+2, y:y+h+7}, color:rgb(0,0,0), thickness:0.25 })
// (나머지 3개 모서리도 동일한 패턴)
```

## 5. 임포지션 프리셋 시스템

### 5.1 프리셋 동작 원리

```
주문 확정 (규격+NUP)  →  프리셋 검색 (sizeCode+nup)  →  프리셋 적용 (자동)  →  PDF 생성 (BullMQ)
```

### 5.2 프리셋 구성 항목

| 항목 | 값 |
|------|-----|
| sizeCode + nup | 복합 유니크 키 (예: "8x10_2up") |
| paperOrientation | landscape / portrait |
| gridCols | NUP 가로 열 수 |
| gridRows | NUP 세로 행 수 |
| marginMm | 크롭마크 수용 여백 (기본 10mm) |
| cropMarkLengthMm | 크롭마크 길이 (기본 5mm) |
| cropMarkThicknessPt | 크롭마크 두께 (기본 0.25pt) |
| cropMarkColor | "K100" 고정 |
| pdfVersion | "1.4" 고정 |
| isActive | 활성화 여부 |

### 5.3 초기 등록 프리셋 (37개 인디고 규격)

| NUP 그룹 | 규격 수 | 대표 규격 |
|---------|---------|----------|
| 16up | 2개 | 3×4, 4×3 |
| 8up | 2개 | 6×4, 4×6 (인디고 전용) |
| 4up | 12개 | 5×7, 7×5, 6×8, 8×6 외 |
| 2up | 14개 | 8×10, 10×8, 11×8, A4_세로, A4_가로 외 |
| 1up | 2개 | 10×10, 11×11 |
| 1+up | 6개 | 13×10, 14×11, 11×14, 15×12 외 |
| 1++up | 1개 | 14×16, 16×14 |

### 5.4 프리셋 없는 규격 처리

- 출력실 화면에서 수동 설정 모드 활성화
- 담당자가 열수/행수/방향 직접 입력 + canvas 미리보기
- 실행 후 "이 설정을 프리셋으로 저장" 옵션 제공
- 저장 시 신규 프리셋 등록 → 다음부터 자동 처리

## 6. DB 스키마 (전체)

OrderItem 1개 확장 + 신규 4개 테이블이 추가됩니다.

### 6.1 OrderItem 확장 (필드 추가)

```prisma
model OrderItem {
  // ... 기존 필드 유지 ...

  // === 출력실 관리 필드 추가 ===
  printRoomStatus     String?    // waiting | ready | imposing | imposed | printing | done
  printRoomReadyAt    DateTime?  // B2 print-ready 준비 완료 시각
  printRoomDoneAt     DateTime?  // 출력완료 시각
  printRoomDoneBy     String?    // 출력완료 처리한 Staff.id

  // === Relations ===
  printReadyFiles     PrintReadyFile[]
  printDownloadLogs   PrintDownloadLog[]
  impositionJobs      ImpositionJob[]
}
```

### 6.2 ImpositionPreset (신규) — 임포지션 프리셋

```prisma
model ImpositionPreset {
  id                  String   @id @default(cuid())
  sizeCode            String   // 규격코드 (예: "8x10")
  nup                 String   // "16up" | "8up" | "4up" | "2up" | "1up" | "1+up" | "1++up"
  paperOrientation    String   // "landscape" | "portrait"
  gridCols            Int      // NUP 열 수
  gridRows            Int      // NUP 행 수
  marginMm            Float    @default(10.0)
  cropMarkLengthMm    Float    @default(5.0)
  cropMarkThicknessPt Float    @default(0.25)
  cropMarkColor       String   @default("K100")
  pdfVersion          String   @default("1.4")
  isActive            Boolean  @default(true)
  createdAt           DateTime @default(now())
  updatedAt           DateTime @updatedAt
  jobs                ImpositionJob[]

  @@unique([sizeCode, nup])
  @@map("imposition_presets")
}
```

### 6.3 ImpositionJob (신규) — 임포지션 처리 이력

```prisma
model ImpositionJob {
  id              String            @id @default(cuid())
  orderItemId     String
  orderItem       OrderItem         @relation(fields: [orderItemId], references: [id])
  presetId        String?
  preset          ImpositionPreset? @relation(fields: [presetId], references: [id])
  isManual        Boolean           @default(false)
  status          String            // pending | processing | done | failed
  pdfB2Key        String?           // 생성된 PDF의 B2 키
  pageCount       Int?              // 총 임포지션 페이지 수
  sourceFileCount Int?              // 원본 JPG 파일 수
  processedAt     DateTime?
  errorLog        String?
  staffId         String?           // 수동 처리 담당자
  createdAt       DateTime          @default(now())

  @@index([orderItemId])
  @@index([status])
  @@map("imposition_jobs")
}
```

### 6.4 PrintReadyFile (신규) — B2 준비 파일

```prisma
model PrintReadyFile {
  id            String              @id @default(cuid())
  orderItemId   String
  orderItem     OrderItem           @relation(fields: [orderItemId], references: [id], onDelete: Cascade)
  b2Key         String              // B2 파일 키
  fileName      String
  fileSize      Int
  printMethod   String              // "inkjet" | "indigo"
  fileType      String              // "jpg" | "pdf"
  preparedAt    DateTime            @default(now())
  downloads     PrintDownloadLog[]

  @@index([orderItemId])
  @@map("print_ready_files")
}
```

### 6.5 PrintDownloadLog (신규) — 핫폴더 전달 이력

```prisma
model PrintDownloadLog {
  id                String          @id @default(cuid())
  orderItemId       String
  orderItem         OrderItem       @relation(fields: [orderItemId], references: [id])
  printReadyFileId  String?
  printReadyFile    PrintReadyFile? @relation(fields: [printReadyFileId], references: [id])
  staffId           String?
  agentId           String?         // 에이전트 ID (자동이면 사용)
  downloadType      String          // "hotfolder_auto" | "manual_download"
  fileCount         Int
  totalBytes        BigInt
  localPath         String?         // 저장된 로컬 경로
  downloadedAt      DateTime        @default(now())

  @@index([orderItemId])
  @@index([downloadedAt])
  @@map("print_download_logs")
}
```

## 8. 출력실 Kanban UI

### 8.1 Kanban 컬럼 구성

| 컬럼 | 상태값 | 의미 | 액션 |
|------|--------|------|------|
| 출력대기 | waiting | 주문 확정, 파일 준비 중 | 자동 진행 |
| 파일준비 | ready | B2 print-ready 복사 완료 | 자동 진행 |
| 임포지션중 | imposing | BullMQ가 PDF 생성 중 | 진행률 표시 |
| 임포지션완료 | imposed | PDF 생성 완료, 핫폴더 대기 | 자동 진행 |
| 출력중 | printing | 핫폴더 전달 완료, RIP 처리 | ✅ 완료 버튼 |
| 완료 | done | 출력 완료 처리됨 | 히스토리로 이동 |

### 8.2 카드 정보

- 주문번호 (orderNumber)
- 스튜디오명 (studioName)
- 규격 + NUP 배지 (예: 8×10 / 2up)
- 페이지수 × 수량 (예: 30p × 3권)
- 출력방식 배지 (인디고 / 잉크젯)
- 현재 상태 + 진행률
- 마지막 처리 담당자 + 시각
- 상태별 액션 버튼

### 8.3 상단 필터 영역

- 날짜 선택 (오늘 / 이번주 / 지정일)
- 출력방식 필터 (전체 / 인디고 / 잉크젯)
- 상태 필터 (활성만 / 전체)
- 실시간 갱신 토글 (자동 새로고침)

## 9. 핫폴더 에이전트

### 9.1 동작 원리

출력 PC에 설치된 에이전트가 30초마다 ERP API에 폴링하여 새 파일이 있으면 B2에서 다운받아 지정된 핫폴더에 저장합니다.

```
에이전트 (30초마다)  →  API 폴링 (GET pending)  →  B2 다운로드 (presigned URL)  →  로컬 저장 (핫폴더)  →  완료 보고 (POST)
```

### 9.2 핫폴더 경로 구조

| 항목 | 값 |
|------|-----|
| 인디고 PDF | `\\printpc\hotfolder\indigo\{YYMMDD}\{orderNum}\` |
| 잉크젯 JPG | `\\printpc\hotfolder\inkjet\{YYMMDD}\{orderNum}\` |
| 에이전트 포트 | http://localhost:9199 |
| 폴링 주기 | 30초 |
| 재시도 정책 | 실패 시 3회, 지수 백오프 |

### 9.3 에이전트 설정 화면

- 자동 핫폴더 ON/OFF 토글
- 저장 경로 변경
- 에이전트 상태 확인 (생존/오류)
- 최근 처리 이력 (마지막 100건)

## 10. B2 저장 구조

### 10.1 폴더 계층

```
photocafe (B2 버킷)
├── originals/                     [기존 — 영구 보관]
│   └── {orderNum}/
│       └── *.jpg                  (고객 업로드 원본)
│
└── print-ready/                   [신규 — 30일 자동 삭제]
    └── {YYMMDD}/
        ├── 인디고/
        │   └── {orderNum}/
        │       └── imposed.pdf    (ERP 임포지션 PDF)
        └── 잉크젯/
            └── {orderNum}/
                └── *.jpg          (원본 복사본)
```

### 10.2 처리 흐름별 B2 저장 위치

| 단계 | 파일 종류 | 저장 위치 | 누가 저장 |
|------|----------|-----------|----------|
| 고객 업로드 | JPG 원본 | `originals/{orderNum}/` | 기존 업로드 모듈 |
| 검수 승인 | (상태만 변경) | DB | OrderFile 모델 |
| 주문 확정 — 잉크젯 | JPG (복사본) | `print-ready/{날짜}/잉크젯/` | B2 copyObject |
| 주문 확정 — 인디고 | 임포지션 PDF | `print-ready/{날짜}/인디고/` | pdf-lib + B2 업로드 |
| 핫폴더 전달 | (B2에서 읽기만) | 출력 PC 로컬 폴더 | 에이전트 |

### 10.3 B2 Lifecycle 수동 설정 (필수)

ERP 코드와 별개로 B2 콘솔에서 한 번 설정해야 합니다.

| 항목 | 값 |
|------|-----|
| Bucket | photocafe |
| File name prefix | `print-ready/` |
| Days from uploading to hiding | 30 |
| Days from hiding to deleting | 0 |

**주의**: `originals/` 폴더는 자동 삭제하면 안 됩니다. 고객 재출력 요청 대응을 위해 영구 보관.

```
에이전트는 Node.js 단독 실행으로 단순하게 구현 (Electron 불필요).
exe 빌드는 pkg 또는 nexe 사용.
```
