# 프론트/백엔드 분리 배포 체크리스트

운영 배포 전 30초 판단용 가이드입니다.

## 1) 프론트만 배포 가능 조건

아래 항목이 모두 `예`이면 프론트만 배포해도 안전합니다.

- [ ] 변경이 UI/문구/스타일/레이아웃 중심이다.
- [ ] 기존 API 엔드포인트(URL, 메서드)가 그대로다.
- [ ] 요청/응답 필드 스키마가 바뀌지 않았다.
- [ ] 인증/권한 규칙(토큰, role, guard)에 변경이 없다.
- [ ] DB 스키마(Prisma 모델/마이그레이션) 변경이 없다.

## 2) 백엔드도 같이 배포해야 하는 조건

하나라도 `예`이면 통합 배포(프론트+백엔드)를 진행하세요.

- [ ] API 경로가 추가/변경/삭제됐다.
- [ ] 응답 JSON 필드명/타입이 바뀌었다.
- [ ] DB 컬럼/테이블/관계가 바뀌었다.
- [ ] 인증 흐름(로그인/리프레시/권한) 로직이 바뀌었다.
- [ ] 업로드/파일 접근/보안 정책이 서버에서 변경됐다.

## 3) 빠른 판단 예시

- `admin-login` 에러 문구/URL fallback 수정: **프론트만 배포 가능**
- 주문 상세 API에 새 필드 추가 + 프론트에서 해당 필드 사용: **통합 배포 필요**
- Prisma 스키마 수정 + 서비스 로직 수정: **통합 배포 필요**

## 4) 권장 실행 순서

1. 변경 파일 확인 (`apps/web`만인지, `apps/api` 포함인지)
2. 위 체크리스트로 분류
3. `프론트만` 또는 `통합` 배포 선택
4. 배포 후 운영 로그인/핵심 화면 1회 스모크 테스트

## 5) 운영 안정성 원칙

- 원칙: **호환성 유지(Backward compatible)**
- 프론트 단독 배포 시, 기존 백엔드 응답과 100% 호환되어야 함
- 불확실하면 통합 배포를 선택
